## What Is LedgerRocket?

LedgerRocket is a real-time core ledger and finance platform for financial services via API.

It centralizes transaction events into a single, strongly consistent, double-entry ledger with an immutable audit trail and matching accounting treatments.

You define your chart of accounts, write declarative templates that describe how business events map to accounting entries, and submit events over the API. LedgerRocket handles the rest -- validating inputs, evaluating template logic, producing double-entry transfers, and writing them atomically to a purpose-built ledger store.

## Who Is It For?

LedgerRocket is designed for banks, fintechs, and regulated institutions:

- **Banks and regulated institutions** that need a compliant, auditable core ledger
- **Fintech companies** building payment products, lending platforms, or money movement infrastructure
- **Neobanks** that need multi-entity, multi-currency ledgering with real-time balances
- **Payment processors** that require high-throughput, idempotent transaction recording
- **FBO (For Benefit Of) providers** managing custodial and beneficial ownership relationships across accounts

## High-Level Architecture

LedgerRocket follows a layered architecture with five stages:

```text
Upstream       Ingestion        Core            Derivations       Consumers
               & Rules          Persistence     & Analytics

+---------+    +-----------+    +-----------+   +------------+    +----------------+
| Product |    | Accounting|    |           |   |            |    | Reconciliation |
| Systems |--->| Rule      |--->| Event     |-->| Data Lake  |--->| Engine         |
| (APIs)  |    | Engine    |    | Store     |   |            |    +----------------+
+---------+    +-----------+    +-----------+   | Balance    |    | GUI            |
               | Event     |    | Journal   |   | Cache      |    +----------------+
               | Service   |--->| Store     |   +------------+    | ERP            |
               +-----------+    | Ledger    |                     +----------------+
               | Reference |    +-----------+                     | MCP / AI       |
               | Data      |                                      | Assistant      |
               | Service   |                                      +----------------+
               +-----------+                                      | Others         |
                                                                  +----------------+
```

| Layer | Components | Purpose |
|-------|-----------|---------|
| **Upstream** | Product systems, external APIs | Source of financial events |
| **Ingestion & Rules** | Accounting Rule Engine, Event Service, Reference Data Service | Validates events, evaluates templates, resolves accounts |
| **Core Persistence** | Event Store, Journal Store Ledger | Immutable storage for events and double-entry transfers |
| **Derivations & Analytics** | Data Lake, Balance Cache | Derived views, real-time balance queries, historical analytics |
| **Consumers** | Reconciliation Engine, GUI, ERP, MCP/AI Assistant | Downstream systems that consume ledger data |

## Core Services

| Service | Responsibility |
|---------|----------------|
| **Reference Data Service** | Manages sites, entities, ledgers, accounts, account codes, currencies, and countries. |
| **Accounting Rule Engine** | Template lifecycle -- authoring, validation, maker-checker approval (DRAFT, PENDING_APPROVAL, LIVE, DEPRECATED). |
| **Event Service** | Accepts financial events, resolves accounts, evaluates templates, produces transfers, and commits them atomically. |
| **Balance Cache** | Real-time and historical balance lookups with caching for immutable historical data. |

## Key Design Principles

### Stateless microservices

Every service instance is stateless. All state lives in the backing stores. Services can scale horizontally without coordination.

### API-first

All services expose OpenAPI 3.1 specifications. Error responses follow RFC 9457 Problem Details. Schemas use JSON Schema 2020-12.

### Idempotent event processing

Every event carries a UUIDv7 `event_id` that serves as an idempotency key. Duplicate submissions are detected and safely ignored. If processing fails mid-flight, an outbox recovery mechanism completes the event automatically.

### Immutable records

Transfers, once written, cannot be modified or deleted. Corrections are made by posting new reversing entries. This provides a complete, auditable history of all financial activity.

### Integer-only math

All monetary amounts are stored and computed as integer minor units (cents, satoshis). There are no floating-point numbers anywhere in the money path. Percentage calculations use basis points with banker's rounding.
